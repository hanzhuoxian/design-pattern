# 状态模式 (State)

## 意图

允许对象在内部状态改变时改变其行为，对象看起来似乎修改了它的类。

## 适用场景

- 对象的行为取决于其状态，且需要在运行时根据状态动态改变行为
- 代码中存在大量基于枚举状态的条件语句（`if/else` 或 `switch`）来切换行为
- 状态转换逻辑复杂，各状态拥有自己的转换规则和副作用
- 希望将各状态的行为封装在独立的类中，便于单独扩展和测试

## 核心结构

```
  ┌───────────────────────────┐
  │       Context (上下文)     │
  │  - state: State           │
  │  + setState(state)        │
  │  + request()              │──────► 委托给当前 state.handle()
  └───────────────────────────┘
                │
                ▼
  ┌───────────────────────────┐
  │       State (接口)         │
  │  + handle(context)        │
  └───────────────────────────┘
            ▲         ▲
            │         │
  ┌──────────────┐ ┌──────────────┐
  │ConcreteStateA│ │ConcreteStateB│
  │+ handle()    │ │+ handle()    │
  │ // 执行A行为  │ │ // 执行B行为  │
  │ // 触发状态转换│ │ // 触发状态转换│
  └──────────────┘ └──────────────┘

状态转换可由 Context 或 ConcreteState 自身发起
```

## 优点

- **符合单一职责原则**：每个状态的行为封装在独立类中，职责清晰
- **符合开闭原则**：新增状态无需修改现有状态类和 Context
- **消除庞大的条件语句**：状态分支逻辑转移到各状态类的多态调用
- **状态转换显式化**：转换规则清晰记录在状态类中，便于追踪

## 缺点

- **状态数量少时过度设计**：简单场景下直接用枚举+条件语句更直观
- **状态类数量较多**：每个状态对应一个类，可能产生大量小类
- **状态间可能相互依赖**：状态 A 需要知道并切换到状态 B，引入隐式耦合
