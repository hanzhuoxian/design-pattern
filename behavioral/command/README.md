# 命令模式 (Command)

## 意图

将请求封装为对象，支持撤销、队列、日志等操作，以及对请求进行参数化。

## 适用场景

- 需要对操作进行参数化，将操作作为参数传递给其他方法或存储在对象中
- 需要实现操作的撤销（Undo）与重做（Redo）功能
- 需要实现操作队列、事务（一组操作原子执行）和延迟执行
- 需要将一系列简单命令组合成复杂宏命令

## 核心结构

```
  ┌──────────────┐         ┌─────────────────────────┐
  │   Invoker    │────────►│     Command (接口)       │
  │  (调用者)    │         │  + execute()             │
  │              │         │  + undo()                │
  └──────────────┘         └─────────────────────────┘
                                        ▲
                                        │
                           ┌────────────────────────┐
                           │   ConcreteCommand       │
                           │  - receiver: Receiver   │
                           │  - savedState           │
                           │  + execute()            │
                           │  + undo()               │
                           └────────────────────────┘
                                        │
                                        ▼
                           ┌────────────────────────┐
                           │      Receiver           │
                           │     (实际执行者)         │
                           │  + action()             │
                           └────────────────────────┘

Client → 创建 ConcreteCommand(receiver) → 传给 Invoker → Invoker.execute()
```

## 优点

- **解耦触发与执行**：Invoker 无需了解 Receiver 的任何信息
- **支持撤销/重做**：在命令对象中保存执行前的状态即可实现
- **可组合成宏命令**：将多个命令组合为一个复合命令，统一调度
- **符合开闭原则**：新增命令无需修改已有的调用者代码
- **支持延迟执行与队列化**：命令对象可以序列化、传输和调度

## 缺点

- **引入额外层次**：在发送者和接收者之间新增了命令类层次，代码量增加
- **类数量膨胀**：每种操作都需要单独的命令类，简单场景下显得冗余
