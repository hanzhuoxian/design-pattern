# 访问者模式 (Visitor)

## 意图

表示作用于对象结构中各元素的操作，在不改变各元素类的前提下定义作用于这些元素的新操作。

## 适用场景

- 需要对复杂对象结构（如抽象语法树 AST、文件系统树）中的所有元素执行某些操作
- 对象结构稳定（元素类型固定），但需要频繁新增作用于这些元素的操作
- 需要在不修改元素类的情况下扩展功能（符合开闭原则）
- 需要对不同类型的元素执行不同的操作，并将相关操作集中管理

## 核心结构

```
  ┌──────────────────────────────────────────┐
  │          Visitor (接口)                   │
  │  + visitConcreteElementA(elem)           │
  │  + visitConcreteElementB(elem)           │
  └──────────────────────────────────────────┘
            ▲                    ▲
            │                    │
  ┌──────────────────┐ ┌──────────────────┐
  │ ConcreteVisitor1 │ │ ConcreteVisitor2 │
  │ + visitElemA()   │ │ + visitElemA()   │
  │ + visitElemB()   │ │ + visitElemB()   │
  └──────────────────┘ └──────────────────┘

  ┌──────────────────────────────┐
  │      Element (接口)           │
  │  + accept(visitor: Visitor)  │
  └──────────────────────────────┘
            ▲              ▲
            │              │
  ┌──────────────┐  ┌──────────────┐
  │ConcreteElemA │  │ConcreteElemB │
  │+ accept(v)   │  │+ accept(v)   │
  │  v.visitA(this)│  v.visitB(this)│
  └──────────────┘  └──────────────┘

双重分派（Double Dispatch）：
  client → elem.accept(visitor) → visitor.visitElemA(this)
```

## 优点

- **符合开闭原则**：新增操作只需新增访问者类，无需修改任何元素类
- **集中相关操作**：同一访问者类中包含了对所有元素类型的处理逻辑，便于维护
- **可累积状态**：访问者在遍历过程中可以积累状态，如统计节点数量、构建结果集

## 缺点

- **新增元素代价高**：每新增一种元素类型，必须修改所有已有访问者类，违反开闭原则
- **破坏封装**：访问者可能需要访问元素的私有成员，导致元素类不得不暴露内部细节
- **适用范围窄**：仅适合元素结构稳定的场景，若元素类型经常变化则维护成本极高
