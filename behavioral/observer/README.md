# 观察者模式 (Observer)

## 意图

定义对象间一对多的依赖关系，当一个对象状态改变时，所有依赖它的对象自动收到通知并更新。

## 适用场景

- 一个对象状态改变需要同时通知其他多个对象（如事件系统、MVC 架构中 Model 通知 View）
- 不希望被观察者（Subject）与观察者（Observer）之间产生紧密耦合
- 观察者的数量和种类在运行时动态变化，不固定
- 需要实现分布式事件处理、消息总线或发布-订阅系统

## 核心结构

```
  ┌──────────────────────────────────────┐
  │         Subject (被观察者)            │
  │  - observers: []Observer             │
  │  + subscribe(observer)               │
  │  + unsubscribe(observer)             │
  │  + notify()                          │
  └──────────────────────────────────────┘
            │ 通知
            ▼
  ┌──────────────────────┐
  │   Observer (接口)     │
  │  + update(event)     │
  └──────────────────────┘
            ▲
     ┌──────┼──────┐
     │      │      │
  ┌──────┐ ┌──────┐ ┌──────┐
  │ ObsA │ │ ObsB │ │ ObsC │
  └──────┘ └──────┘ └──────┘

Subject 状态变化 → notify() → 依次调用每个 Observer.update()
```

## 优点

- **符合开闭原则**：新增观察者无需修改被观察者代码
- **运行时动态建立联系**：可以在运行时随时订阅或取消订阅
- **被观察者无需了解观察者的具体类型**：依赖抽象接口，充分解耦

## 缺点

- **通知顺序不确定**：多个观察者的执行顺序可能不可预期
- **级联更新风险**：观察者的 `update` 可能触发另一个 Subject 的变化，引发意外的链式通知
- **内存泄漏隐患**：若观察者注册后未及时取消订阅，Subject 会持有其引用，阻止对象被回收
