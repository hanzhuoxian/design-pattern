# 责任链模式 (Chain of Responsibility)

## 意图

使多个对象都有机会处理请求，将这些对象连成一条链，沿链传递请求直到有对象处理为止。

## 适用场景

- 不确定哪个对象会处理请求时（如审批流程、HTTP 中间件管道）
- 需要以特定顺序依次执行多个处理者
- 处理者集合需要在运行时动态变化或重新组合
- 希望发出请求的对象与处理请求的对象解耦

## 核心结构

```
     Client
       │
       ▼
  ┌─────────────────────────────────────┐
  │          Handler (接口/抽象类)       │
  │  + setNext(handler): Handler        │
  │  + handle(request): response        │
  └─────────────────────────────────────┘
       ▲               ▲               ▲
       │               │               │
┌──────────────┐ ┌──────────────┐ ┌──────────────┐
│ConcreteHandler│ │ConcreteHandler│ │ConcreteHandler│
│      A       │►│      B       │►│      C       │
│              │ │              │ │              │
│ - next       │ │ - next       │ │ - next(nil)  │
└──────────────┘ └──────────────┘ └──────────────┘

请求沿链传递：A → B → C（直到某个处理者消费请求）
```

## 优点

- **控制处理顺序**：可灵活调整链中各处理者的顺序
- **符合单一职责原则**：每个处理者只关注自己能处理的请求类型
- **符合开闭原则**：新增处理者无需修改现有代码
- **可动态组合**：运行时可以按需组装、拆解处理链

## 缺点

- **请求可能未被处理**：若链中无处理者匹配，请求会被静默忽略（需额外兜底逻辑）
- **调试困难**：链路较长时难以追踪请求到底经过了哪些节点
- **性能开销**：请求可能需要穿越整条链才能找到处理者
