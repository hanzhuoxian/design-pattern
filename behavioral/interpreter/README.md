# 解释器模式 (Interpreter)

## 意图

给定一个语言，定义它的文法表示，并定义一个解释器来解释该语言中的句子。

## 适用场景

- 需要解释特定领域语言的句子（如 SQL 解析、正则表达式、脚本语言、规则引擎）
- 文法相对简单，且该语言的句子需要频繁被解释和执行
- 需要将语法规则表示为类层次结构，使文法易于扩展和维护
- 性能不是首要考量，可以接受解释执行带来的开销

## 核心结构

```
  ┌──────────────────────────────────┐
  │     AbstractExpression (接口)    │
  │  + interpret(context): result    │
  └──────────────────────────────────┘
            ▲                  ▲
            │                  │
  ┌──────────────────┐  ┌───────────────────────────┐
  │TerminalExpression│  │  NonTerminalExpression      │
  │   (终结符表达式)  │  │    (非终结符表达式)          │
  │                  │  │  - left: AbstractExpression │
  │ + interpret()    │  │  - right: AbstractExpression│
  └──────────────────┘  │  + interpret()              │
                        └───────────────────────────┘

  Context (上下文)：存储解释器所需的全局信息

  示例句子：A AND (B OR C)
  解析树：
       AND
      /   \
     A    OR
         /  \
        B    C
```

## 优点

- **易于改变和扩展文法**：每条规则对应一个类，新增规则只需新增类
- **实现文法简单**：每个类的 `interpret` 方法通常代码量少，逻辑清晰
- **可复合**：通过组合终结符和非终结符表达式构造任意复杂的句子

## 缺点

- **复杂文法难以维护**：规则一多，类数量急剧膨胀，整体结构难以掌握
- **执行效率较低**：递归调用、大量对象创建会影响性能，不适合高性能场景
- **不适合复杂语法**：通常更适合用专门的解析器生成器（如 ANTLR、yacc）处理
