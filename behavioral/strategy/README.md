# 策略模式 (Strategy)

## 意图

定义一系列算法，将其封装并可相互替换，使算法的变化独立于使用算法的客户。

## 适用场景

- 需要在运行时切换不同的算法或行为（如排序策略、支付方式、数据压缩算法）
- 存在大量仅在执行某一行为时略有不同的相似类，需要消除重复
- 存在大量条件语句来选择不同算法分支，希望用多态替代条件判断
- 算法实现细节与使用算法的业务代码需要隔离，使两者可以独立变化

## 核心结构

```
  ┌──────────────────────────────┐
  │        Context (上下文)       │
  │  - strategy: Strategy        │
  │  + setStrategy(strategy)     │
  │  + executeStrategy()         │──────► 委托给 strategy.execute()
  └──────────────────────────────┘

  ┌──────────────────────────────┐
  │      Strategy (接口)          │
  │  + execute(data): result     │
  └──────────────────────────────┘
            ▲           ▲
            │           │
  ┌──────────────┐ ┌──────────────┐
  │ ConcreteStratA│ │ ConcreteStratB│
  │ + execute()  │ │ + execute()  │
  │ // 算法实现A  │ │ // 算法实现B  │
  └──────────────┘ └──────────────┘

Client → context.setStrategy(new StrategyB()) → context.executeStrategy()
```

## 优点

- **运行时切换算法**：可在不修改 Context 的情况下，随时替换使用的算法
- **隔离实现细节**：算法实现与调用代码完全分离，各自独立演化
- **符合开闭原则**：新增算法只需实现新的策略类，无需修改 Context
- **消除条件分支**：用多态替代 `if/switch` 算法选择，代码更清晰

## 缺点

- **客户端需了解不同策略**：客户端必须知道各策略的区别才能做出正确选择
- **函数式替代方案更轻量**：支持一等函数的语言通常可以用函数/闭包替代策略类，无需定义接口和类
- **对象数量增加**：每种算法对应一个策略类，简单场景下类数量增多
