# 备忘录模式 (Memento)

## 意图

在不破坏封装性的前提下，捕获对象的内部状态并在外部保存，以便在需要时将对象恢复到之前的状态。

## 适用场景

- 需要实现撤销（Undo）操作，如文本编辑器、图形编辑器、游戏存档
- 直接访问对象字段会暴露实现细节，破坏封装性
- 需要生成对象在某个时刻的状态快照（Snapshot）
- 希望将状态历史管理的责任从原发器中分离出来

## 核心结构

```
  ┌──────────────────────────┐         ┌────────────────────────┐
  │     Originator (原发器)   │         │    Memento (备忘录)     │
  │  - state                 │────────►│  - state (不透明)       │
  │  + createMemento()       │         │  + getState()           │
  │  + restore(memento)      │         └────────────────────────┘
  └──────────────────────────┘                   ▲
                                                 │ 存储/取回
  ┌──────────────────────────────────────────────┐
  │             Caretaker (负责人)                │
  │  - history: []Memento                        │
  │  + save()    // 调用 originator.createMemento │
  │  + undo()    // 调用 originator.restore       │
  └──────────────────────────────────────────────┘

Caretaker 只负责存储备忘录，不检查也不修改其内容（不透明）
Originator 是唯一知道如何创建和恢复备忘录的对象
```

## 优点

- **不破坏封装**：外部只持有不透明的备忘录对象，无法直接访问原发器的内部状态
- **简化原发器**：原发器无需自行管理历史状态栈，职责更单一
- **恢复机制简洁**：状态恢复逻辑集中在原发器内部，清晰可控

## 缺点

- **内存消耗大**：若状态体积大或频繁创建快照，会占用大量内存
- **Caretaker 需关注生命周期**：需要知道何时可以安全地丢弃旧备忘录，避免内存泄漏
- **封装强度依赖语言特性**：备忘录的"不透明性"依赖语言的访问控制机制，不同语言的实现效果存在差异
