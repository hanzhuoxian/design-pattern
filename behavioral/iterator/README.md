# 迭代器模式 (Iterator)

## 意图

提供一种方法顺序访问聚合对象中各个元素，而不暴露其内部表示。

## 适用场景

- 需要遍历不同类型的集合（数组、树、图、链表），同时对调用者隐藏其内部复杂性
- 需要同时存在多个遍历同一集合的迭代器，各自维护独立的遍历状态
- 需要统一遍历不同集合类型的接口，使客户端代码与具体集合解耦
- 需要支持惰性求值（只在需要时才计算下一个元素，如处理大数据流）

## 核心结构

```
  ┌─────────────────────┐         ┌─────────────────────┐
  │   Iterable (接口)   │         │   Iterator (接口)    │
  │  + iterator()       │────────►│  + hasNext(): bool   │
  └─────────────────────┘         │  + next(): Element   │
            ▲                     └─────────────────────┘
            │                               ▲
  ┌─────────────────────┐         ┌─────────────────────┐
  │  ConcreteCollection  │         │  ConcreteIterator    │
  │  - items[]          │         │  - collection        │
  │  + iterator()       │────────►│  - position: int     │
  └─────────────────────┘         │  + hasNext()         │
                                  │  + next()            │
                                  └─────────────────────┘

Client → collection.iterator() → 得到 Iterator → 逐步调用 next()
```

## 优点

- **单一职责原则**：将遍历逻辑从集合中提取出来，各司其职
- **可并行迭代**：多个迭代器可以同时遍历同一集合，互不干扰
- **可暂停与恢复**：迭代器保存遍历状态，可以随时中断并在之后继续
- **统一接口**：不同结构的集合可以用同一套迭代接口访问

## 缺点

- **过度设计风险**：对简单数组等集合直接遍历更高效，引入迭代器显得繁重
- **某些集合效率低**：特殊数据结构（如跳表、B 树）直接遍历效率更高
