# 工厂方法模式 (Factory Method Pattern)

## 意图

定义一个创建对象的接口，但由子类决定实例化哪个类，从而将对象的实例化延迟到子类。

## 适用场景

- **创建对象的代码无法预知具体类**：在编写框架时，框架知道何时需要创建对象，但不知道应创建哪种对象（由使用者决定）
- **希望子类指定创建的对象**：允许框架的使用者通过子类化来扩展内部组件，而无需修改框架本身
- **复用现有对象创建逻辑**：将产品创建代码集中到一处，避免散落在各处，便于维护和复用
- **需要对产品进行统一初始化**：在工厂方法中可以添加日志、缓存、权限检查等横切逻辑

## 核心结构

```
┌─────────────────────────────┐         ┌──────────────────────┐
│        Creator              │         │       Product        │
│        (抽象类)             │         │       (接口)         │
├─────────────────────────────┤         ├──────────────────────┤
│ + factoryMethod(): Product  │─ ─ ─ ─→│ + operation()        │
│ + someOperation()           │         └──────────────────────┘
└──────────────┬──────────────┘                    ▲
               │ 继承                              │ 实现
               ▼                                   │
┌─────────────────────────────┐         ┌──────────────────────┐
│      ConcreteCreator        │         │   ConcreteProduct    │
├─────────────────────────────┤         ├──────────────────────┤
│ + factoryMethod(): Product  │────────→│ + operation()        │
│   // 返回 ConcreteProduct   │  创建   └──────────────────────┘
└─────────────────────────────┘

调用流程：
Client ──→ creator.someOperation()
                   │
                   ▼
           调用 factoryMethod()  ← 由子类实现，返回具体产品
                   │
                   ▼
           使用 product.operation() 完成业务逻辑
```

## 优点

- **避免创建者与具体产品的紧耦合**：创建者无需知道将要创建的产品的具体类型，依赖抽象而非具体实现
- **符合单一职责原则**：将产品创建代码集中到程序的单一位置，使代码更容易维护
- **符合开闭原则**：无需修改现有代码即可引入新的产品类型，只需新增具体创建者子类

## 缺点

- **类数量增多**：每新增一种产品，就需要新增对应的创建者子类，可能导致类层次结构变得庞大
- **继承带来的复杂性**：客户端需要子类化创建者，如果创建者类已经有子类层次，情况会更加复杂

## 延伸阅读

- 工厂方法模式常与模板方法模式配合使用：模板方法定义算法框架，工厂方法负责创建中间步骤所需的对象
- 与简单工厂（Simple Factory）的区别：简单工厂不是 GoF 设计模式，它将所有创建逻辑集中在一个类中，不符合开闭原则
- 不依赖继承的语言通常通过函数类型或接口来实现工厂方法，具体方式详见各语言实现目录
- 抽象工厂模式可以视为工厂方法模式的扩展，用于创建一族相关产品
