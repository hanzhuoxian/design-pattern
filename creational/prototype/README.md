# 原型模式 (Prototype Pattern)

## 意图

通过复制（克隆）现有对象来创建新对象，而非重新实例化，使得对象的创建与其具体类解耦。

## 适用场景

- **对象初始化成本高**：对象的创建涉及大量计算、数据库查询或网络请求时，克隆已有对象比重新初始化代价更低
- **需要在运行时指定实例化的类**：当需要创建的对象类型在编译期未知，只能在运行时通过已有对象克隆获得
- **对象只有几种状态组合，克隆比手工配置更方便**：预先配置好几个"原型"对象，按需克隆并微调，比每次从头构建更高效
- **需要独立于其产品的类层次结构**：克隆不依赖具体类，只依赖克隆接口，适合插件化或动态扩展场景

## 核心结构

```
┌──────────────────────────────┐
│         Prototype            │
│         (接口)               │
├──────────────────────────────┤
│ + clone(): Prototype         │
└──────────────┬───────────────┘
               │ 实现
       ┌───────┴────────┐
       ▼                ▼
┌──────────────┐  ┌──────────────┐
│ ConcreteP1   │  │ ConcreteP2   │
├──────────────┤  ├──────────────┤
│ - field1     │  │ - fieldA     │
│ - field2     │  │ - fieldB     │
├──────────────┤  ├──────────────┤
│ + clone()    │  │ + clone()    │
│   // 复制自身│  │   // 复制自身│
└──────────────┘  └──────────────┘

克隆流程：
Client ──→ prototype.clone()
                   │
                   ▼
           创建同类型新对象
                   │
         ┌─────────┴──────────┐
         ▼                    ▼
     浅拷贝（Shallow）    深拷贝（Deep）
     值类型字段直接复制   引用类型字段递归复制
     引用类型字段共享     确保完全独立
         │                    │
         └─────────┬──────────┘
                   ▼
             返回新对象（与原型相互独立）
```

## 优点

- **不耦合到具体类即可克隆**：客户端只需知道克隆接口，无需了解被克隆对象的具体类型，降低依赖
- **避免重复初始化**：对于初始化成本高的对象，克隆比重新构建性能更优，尤其适合对象池场景
- **运行时动态创建对象**：可在运行时注册和更换原型，灵活应对动态变化的需求，无需修改代码
- **替代继承实现对象变体**：通过预置不同状态的原型对象，避免构建庞大的子类层次

## 缺点

- **克隆含循环引用的复杂对象较棘手**：若对象图中存在循环引用，克隆时需要额外的追踪机制（如已克隆对象注册表）来避免无限递归
- **深拷贝实现复杂**：需要确保所有嵌套引用类型字段都被正确复制，否则克隆体与原型会共享内部状态，导致意料之外的副作用
- **每个类都需实现克隆接口**：对于已有的类层次结构，若不支持克隆，需要逐一改造，改造成本较高

## 延伸阅读

- 原型注册表（Prototype Registry）：维护一个命名原型的字典，客户端通过名称检索原型并克隆，是原型模式的常见扩展
- 与工厂方法的比较：工厂方法通过子类化决定创建何种对象，原型则通过克隆已有对象来创建，无需子类化
- 大多数语言没有通用的深拷贝机制，通常需要手动实现克隆方法，或借助序列化/反序列化实现，具体做法详见各语言实现目录
- 部分语言将原型思想内嵌于语言特性本身，无需显式引入此模式
