# 建造者模式 (Builder Pattern)

## 意图

将一个复杂对象的构建过程与其表示分离，使得同样的构建过程可以创建不同的表示。

## 适用场景

- **创建复杂对象需要多个步骤**：如 SQL 查询构建器、HTML 生成器、HTTP 请求构建器，每一步都是独立的配置操作
- **不同配置产生不同对象表示**：同一套构建流程，通过切换不同的建造者实现，可得到格式迥异的产品（如将同一份数据导出为 JSON 或 XML）
- **需要构建过程与产品表示解耦**：Director 只负责控制构建顺序，不关心产品的最终形态，便于在不修改 Director 的前提下替换建造者
- **避免构造函数参数爆炸**：当对象有大量可选参数时，建造者可以替代"重叠构造函数（Telescoping Constructor）"，使代码更具可读性

## 核心结构

```
┌─────────────────────┐         ┌──────────────────────────────┐
│      Director       │         │        Builder               │
├─────────────────────┤         │        (接口)                │
│ - builder: Builder  │────────→├──────────────────────────────┤
├─────────────────────┤  使用   │ + reset()                    │
│ + construct()       │         │ + buildStepA()               │
└─────────────────────┘         │ + buildStepB()               │
                                │ + buildStepC()               │
                                │ + getResult(): Product       │
                                └──────────────┬───────────────┘
                                               │ 实现
                                    ┌──────────┴──────────┐
                                    ▼                     ▼
                          ┌──────────────────┐  ┌──────────────────┐
                          │ ConcreteBuilder1 │  │ ConcreteBuilder2 │
                          ├──────────────────┤  ├──────────────────┤
                          │ 构建 ProductA    │  │ 构建 ProductB    │
                          └──────────────────┘  └──────────────────┘

调用流程：
Client ──→ 创建 ConcreteBuilder
        ──→ 创建 Director(builder)
        ──→ director.construct()
                │
                ├── builder.reset()
                ├── builder.buildStepA()
                ├── builder.buildStepB()
                └── builder.buildStepC()
        ──→ builder.getResult() → 获取最终产品
```

## 优点

- **可分步骤构建对象**：构建过程可以暂停、延迟，甚至递归执行，灵活性远超一步到位的构造函数
- **可复用构建代码生成不同产品**：同一个 Director 配合不同的 Builder 实现，可生成外观迥异的产品，避免代码重复
- **符合单一职责原则**：复杂的构建逻辑从业务代码中分离到独立的建造者类中，各司其职

## 缺点

- **产品需要有通用接口**：若不同建造者构建出来的产品差异过大，难以抽象出共同接口，模式的适用性降低
- **代码整体复杂度增加**：引入了 Director、Builder 接口、多个 ConcreteBuilder 等额外类，对于简单对象是过度设计
- **Director 并非必须**：在许多实际应用中（如链式调用风格的 Builder），Director 会被省略，由客户端直接控制构建顺序

## 延伸阅读

- 建造者模式在现代语言中常以"链式调用（Method Chaining）"风格出现，如 `QueryBuilder.select("*").from("users").where("age > 18").build()`
- 与抽象工厂的区别：抽象工厂一次性返回产品族，建造者则是逐步构建单个复杂产品
- 许多语言提供了建造者的轻量替代方案（如命名参数、可选参数、函数选项等），适合参数较多但结构简单的场景
- 建造者模式在标准库和第三方库中十分常见，HTTP 客户端、进程启动、查询构建等场景尤为典型
