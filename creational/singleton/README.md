# 单例模式 (Singleton Pattern)

## 意图

保证一个类仅有一个实例，并提供一个全局访问点。

## 适用场景

- **全局配置管理**：应用程序配置信息只需加载一次，全局共享
- **连接池**：数据库连接池、线程池等资源池需要统一管理，避免重复创建
- **日志记录器**：日志系统通常需要一个统一的入口，保证日志写入的一致性
- **线程池**：管理线程资源，避免无限制地创建线程导致系统资源耗尽

## 核心结构

```
┌─────────────────────────────────────────┐
│              Singleton                  │
├─────────────────────────────────────────┤
│ - instance: Singleton  (私有静态字段)    │
├─────────────────────────────────────────┤
│ - Singleton()          (私有构造函数)    │
│ + getInstance(): Singleton              │
└─────────────────────────────────────────┘
         ▲
         │ 调用 getInstance()
         │
┌────────┴────────┐
│     Client      │
└─────────────────┘

调用流程：
Client ──→ Singleton.getInstance() ──→ 检查 instance 是否为 nil
                                              │
                              ┌───────────────┴───────────────┐
                           是 nil                           不是 nil
                              │                               │
                         创建新实例                       返回已有实例
                              └───────────────┬───────────────┘
                                              ▼
                                       返回唯一实例
```

## 优点

- **控制实例数量**：严格控制客户端如何访问唯一实例，防止外部代码直接覆盖
- **节省资源**：对于创建成本较高的对象（如数据库连接），避免重复创建，节省系统资源
- **提供全局访问点**：类似全局变量，但更安全，不允许外部代码覆盖，且支持懒加载

## 缺点

- **违反单一职责原则**：该模式同时解决了两个问题（保证唯一实例 + 提供全局访问点）
- **难以单元测试**：单例的私有构造函数和全局状态使得 mock 和隔离测试变得困难
- **多线程需额外处理**：在并发环境下，若无保护措施可能创建多个实例，需使用双重检查锁（Double-Checked Locking）或语言提供的原子初始化机制
- **隐藏依赖关系**：使用单例的代码隐式依赖全局状态，降低代码可读性和可维护性

## 使用示例

以**配置管理器**为例：应用启动时需要从文件或环境变量中读取一次配置（数据库地址、日志级别、端口号等），后续所有模块都应读取同一份，而不是各自重复加载。

- 首次调用 `GetConfig()` 时，执行加载逻辑并将结果保存为唯一实例
- 后续任意模块再次调用时，直接返回已有实例，不重复执行加载
- 并发调用时，语言或框架提供的原子初始化机制保证只有一个 goroutine/线程执行加载

其他典型场景（日志记录器、连接池）遵循相同逻辑：用一个全局访问函数封装"首次创建、后续复用"的过程。

## 延伸阅读

- 单例模式是最简单的设计模式之一，但在实际使用中需警惕滥用
- 如果只是需要一个"方便访问的全局对象"，可以考虑依赖注入（DI）作为替代
- 线程安全的单例通常有两种策略：饿汉式（程序启动时即初始化）和懒汉式（首次访问时初始化）
- 各语言通常提供原生的原子初始化原语，优先使用它们而非手写锁逻辑
