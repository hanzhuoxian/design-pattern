# 抽象工厂模式 (Abstract Factory Pattern)

## 意图

提供一个创建一系列相关或相互依赖对象的接口，而无需指定它们的具体类。

## 适用场景

- **系统需要独立于产品的创建、组合与表示**：当业务逻辑不应依赖具体的产品实现时，通过抽象工厂隔离变化
- **需要配置多个产品族**：典型场景如跨平台 UI 组件库（Windows 风格 vs macOS 风格），运行时只需切换工厂，无需修改业务代码
- **需要确保同族产品的兼容性**：如深色主题的按钮只应与深色主题的文本框搭配，抽象工厂从结构上保证产品族内部一致性
- **需要对外隐藏产品的具体实现**：对外只暴露接口，内部实现可随时替换，无需修改调用方代码

## 核心结构

```
┌───────────────────────────────────┐
│         AbstractFactory           │
│           (接口)                  │
├───────────────────────────────────┤
│ + createProductA(): AbstractA     │
│ + createProductB(): AbstractB     │
└──────────────┬────────────────────┘
               │ 实现
       ┌───────┴────────┐
       ▼                ▼
┌──────────────┐  ┌──────────────┐
│ ConcreteF1   │  │ ConcreteF2   │
├──────────────┤  ├──────────────┤
│createProductA│  │createProductA│  ← 各自创建同族产品
│createProductB│  │createProductB│
└──────┬───────┘  └──────┬───────┘
       │                 │
  创建族1产品         创建族2产品
       │                 │
       ▼                 ▼
┌──────────────┐  ┌──────────────┐
│ ProductA1    │  │ ProductA2    │  ← 实现 AbstractA
├──────────────┤  ├──────────────┤
│ ProductB1    │  │ ProductB2    │  ← 实现 AbstractB
└──────────────┘  └──────────────┘

同族产品保证兼容：
  工厂1 → { ProductA1 + ProductB1 }  ✅ 兼容
  工厂2 → { ProductA2 + ProductB2 }  ✅ 兼容
  混用  → { ProductA1 + ProductB2 }  ❌ 不兼容
```

## 优点

- **确保同族产品的兼容性**：通过工厂约束，杜绝了不同产品族之间的错误组合，降低运行时错误风险
- **符合开闭原则**：新增产品族时只需新增一个具体工厂类，无需修改已有代码
- **将产品创建集中管理**：产品的实例化细节被封装在工厂中，调用方无需了解具体实现，降低耦合

## 缺点

- **新增产品种类困难**：如果需要在现有产品族中新增一种产品（如新增 `createProductC`），则需要修改抽象工厂接口及所有具体工厂实现，违反开闭原则
- **代码复杂度较高**：引入大量接口和类，对于产品族较少的场景可能过度设计

## 延伸阅读

- 抽象工厂与工厂方法的关系：抽象工厂通常由多个工厂方法组成，可以视为"工厂的工厂"
- 典型应用：数据库驱动切换（MySQL/PostgreSQL）、UI 主题系统、跨平台组件库
- 不支持类继承的语言通常通过接口组合或泛型约束来模拟抽象工厂，具体方式详见各语言实现目录
