# 桥接模式 (Bridge Pattern)

## 意图

将抽象部分与实现部分分离，使二者可以独立变化。

---

## 适用场景

- 希望在多个维度上扩展类（例如形状 × 颜色），避免类数量爆炸
- 需要在运行时切换不同的实现，而无需修改抽象层
- 多个不同的抽象需要共享同一套实现逻辑
- 需要在不同平台上运行，且希望实现层能独立替换

---

## 核心结构

```
Abstraction ──────────────► Implementor (接口)
     │  持有 impl 引用           ▲
     │                     ┌────┴────┐
     ▼                     │         │
RefinedAbstraction   ConcreteImplA  ConcreteImplB
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Abstraction` | 定义高层控制逻辑，持有 `Implementor` 引用 |
| `RefinedAbstraction` | 扩展 `Abstraction`，提供更精细的控制变体 |
| `Implementor` | 声明所有实现类通用的接口 |
| `ConcreteImplementor` | 提供平台特定的实现 |

---

## 优点

- **独立扩展**：抽象层与实现层可各自独立添加新变体，互不影响
- **开闭原则**：引入新抽象或新实现无需修改现有代码
- **运行时切换实现**：可以在程序运行期间动态替换实现对象
- **消除类爆炸**：避免多继承维度导致子类数量指数级增长

---

## 缺点

- 对高内聚的类使用桥接模式会增加不必要的复杂度
- 需要在设计阶段提前识别出两个独立变化的维度，否则适用性降低
- 增加了一层间接调用，对性能极敏感场景需评估
