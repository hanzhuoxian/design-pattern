# 组合模式 (Composite Pattern)

## 意图

将对象组合成树形结构以表示"部分-整体"层次，使客户端对单个对象和组合对象的使用具有一致性。

---

## 适用场景

- 需要实现树状对象结构，例如文件系统、UI 组件树、组织架构图
- 希望客户端统一处理简单元素（叶节点）和复杂元素（容器），无需区分二者
- 需要对树中任意节点执行相同操作（如递归统计、渲染、权限校验）
- 结构可以任意嵌套，且层级深度在编译期未知

---

## 核心结构

```
        Component (接口/抽象类)
        operation()
        add() / remove() / getChild()
           ▲              ▲
           │              │
         Leaf          Composite
       operation()    operation()
                      children: Component[]
                      (递归调用所有子节点)
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Component` | 声明叶节点和容器共同拥有的接口 |
| `Leaf` | 树的基本元素，无子节点，执行实际工作 |
| `Composite` | 有子节点的容器，将操作委托给子节点并汇总结果 |
| `Client` | 通过 `Component` 接口与所有元素交互 |

---

## 优点

- **统一处理**：利用多态和递归，客户端无需区分叶节点和容器，代码更简洁
- **开闭原则**：引入新的元素类型无需修改现有代码
- **自然表达层次结构**：树形数据天然适合此模式，语义清晰

---

## 缺点

- 对功能差异较大的类提供公共接口较困难，可能导致接口过于笼统
- 设计可能过度泛化，某些只对特定类型节点有意义的操作难以约束
- 某些语言难以在编译期通过类型系统区分叶节点和容器（需运行时判断）
