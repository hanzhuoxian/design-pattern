# 结构型模式 (Structural Patterns)

结构型模式关注类和对象的**组合方式**。它们通过组合多个类或对象，形成更大的结构，同时保持结构的灵活性与高效性，使系统更易于扩展和维护。

## 核心思想

- 通过组合而非继承来扩展功能，降低类之间的耦合
- 用简单对象组合出复杂结构，同时对外保持统一接口
- 在不改变已有类的前提下，为其附加新能力或适配新接口

## 模式一览

| 模式 | 意图 | 核心关键词 |
|------|------|-----------|
| [适配器模式](./adapter/README.md) | 将一个类的接口转换为客户端期望的另一个接口 | 接口转换、兼容性 |
| [桥接模式](./bridge/README.md) | 将抽象部分与实现部分分离，使两者可以独立变化 | 抽象与实现分离、多维度扩展 |
| [组合模式](./composite/README.md) | 将对象组合成树形结构，使单个对象和组合对象的使用方式一致 | 树形结构、统一接口 |
| [装饰器模式](./decorator/README.md) | 动态地为对象添加额外职责，比继承更灵活 | 动态扩展、包装 |
| [外观模式](./facade/README.md) | 为子系统提供统一的高层接口，简化客户端调用 | 简化接口、封装子系统 |
| [享元模式](./flyweight/README.md) | 共享细粒度对象以减少内存消耗 | 共享、内存优化 |
| [代理模式](./proxy/README.md) | 为对象提供代理以控制对其的访问 | 访问控制、延迟初始化 |

## 如何选择

```
需要处理类或对象的组合结构
    │
    ├─ 需要让不兼容的接口协同工作？
    │       └─ 是 → 适配器模式
    │
    ├─ 有两个独立变化的维度，避免类爆炸？
    │       └─ 是 → 桥接模式
    │
    ├─ 需要以统一方式处理单个对象和对象集合？
    │       └─ 是 → 组合模式
    │
    ├─ 需要在运行时动态为对象添加功能，不想修改原类？
    │       └─ 是 → 装饰器模式
    │
    ├─ 需要简化复杂子系统的调用，提供统一入口？
    │       └─ 是 → 外观模式
    │
    ├─ 需要大量相似对象，内存占用是瓶颈？
    │       └─ 是 → 享元模式
    │
    └─ 需要控制对对象的访问（懒加载、权限、缓存等）？
            └─ 是 → 代理模式
```
