# 代理模式 (Proxy Pattern)

## 意图

为其他对象提供代理以控制对这个对象的访问。

---

## 适用场景

- **延迟初始化（虚拟代理）**：真实对象初始化代价高昂，仅在真正需要时才创建
- **访问控制（保护代理）**：根据调用方权限决定是否允许访问真实对象
- **远程服务本地执行（远程代理）**：将远程服务调用封装成本地对象，隐藏网络细节
- **请求记录（日志代理）**：在访问真实对象前后记录请求历史或审计信息
- **缓存代理**：对开销较大的操作结果进行缓存，多个客户端可共享同一结果

---

## 核心结构

```
Client ──► Subject (接口)
               ▲              ▲
               │              │
          RealSubject        Proxy
         （真实对象）    realSubject: RealSubject ◄── 持有引用
                         （控制访问、懒加载、缓存等）
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Subject` | 声明真实对象和代理的公共接口，客户端通过此接口与二者交互 |
| `RealSubject` | 包含核心业务逻辑的真实对象 |
| `Proxy` | 持有对真实对象的引用，控制对其的访问，可在访问前后执行额外逻辑 |
| `Client` | 通过 `Subject` 接口与代理或真实对象交互，无需感知二者的区别 |

**常见代理类型对比：**

| 代理类型 | 核心用途 | 典型场景 |
|---------|---------|---------|
| 虚拟代理 | 延迟创建开销大的对象 | 大图片懒加载 |
| 保护代理 | 权限校验，控制访问 | 角色权限系统 |
| 远程代理 | 本地代表远程对象 | RPC、gRPC Stub |
| 日志代理 | 记录操作日志 | 审计、调试 |
| 缓存代理 | 缓存昂贵操作结果 | HTTP 缓存、查询缓存 |

---

## 优点

- **客户端无感知**：客户端可在不感知的情况下控制服务对象的访问行为
- **开闭原则**：可以引入新的代理而无需修改服务或客户端代码
- **生命周期管理**：代理可在客户端不关心时管理服务对象的生命周期
- **懒加载支持**：服务对象在首次使用时才被创建，节省资源

---

## 缺点

- 引入额外间接层，代码复杂度增加
- 代理层可能引入响应延迟，性能敏感场景需评估
- 需要维护与真实对象相同的接口，接口变更时代理也需同步更新
