# 装饰器模式 (Decorator Pattern)

## 意图

动态地给对象添加额外职责，比生成子类更为灵活。

---

## 适用场景

- 在不修改原有代码的情况下为对象添加行为，遵循开闭原则
- 需要通过组合而非继承扩展功能，避免子类数量爆炸
- 需要在运行时组合多种行为，且不同对象可能使用不同的行为组合
- 功能可以从对象上"撤销"——通过移除装饰器而非修改类

---

## 核心结构

```
        Component (接口)
        operation()
           ▲              ▲
           │              │
   ConcreteComponent    Decorator
     operation()        operation()
                        wrappee: Component  ◄─── 持有被装饰对象的引用
                            ▲
                            │
                   ConcreteDecoratorA / ConcreteDecoratorB
                   operation()（调用 super + 额外行为）
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Component` | 声明装饰器和被装饰对象的公共接口 |
| `ConcreteComponent` | 定义基础行为，可被装饰器包裹 |
| `Decorator` | 抽象装饰器，持有 `Component` 引用，将操作委托给包裹对象 |
| `ConcreteDecorator` | 在委托前后添加额外行为 |

---

## 优点

- **无需创建子类即可扩展行为**：功能通过组合堆叠而非继承实现
- **运行时组合**：可以在运行时为同一对象按需添加或移除装饰器
- **单一职责原则**：每个装饰器专注于单一功能，职责清晰

---

## 缺点

- 在调用栈特定层移除某个特定的包装器较困难
- 装饰器的顺序会影响最终行为，顺序管理需谨慎
- 调试时调用栈层数多，较难追踪问题根源
- 实现大量小型装饰器类时，代码量增加明显
