# 外观模式 (Facade Pattern)

## 意图

为子系统中的一组接口提供一个统一的高层接口，使子系统更容易使用。

---

## 适用场景

- 为复杂子系统提供简单的入口，屏蔽底层细节
- 对客户端隐藏子系统的内部复杂性，降低使用门槛
- 需要对子系统进行分层，以减少各层之间的耦合
- 遗留系统迁移时，通过外观层逐步替换旧实现

---

## 核心结构

```
Client
  │
  ▼
Facade（统一入口，协调各子系统）
  │
  ├──► SubsystemA.operationA()
  ├──► SubsystemB.operationB()
  └──► SubsystemC.operationC()
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Facade` | 提供简单接口，知道哪些子系统负责处理请求，并将其委托给相应对象 |
| `Subsystem` | 实现子系统功能，处理由 `Facade` 分配的工作，不感知 `Facade` 的存在 |
| `Client` | 通过 `Facade` 与子系统交互，而非直接调用子系统 |

---

## 优点

- **隔离复杂性**：客户端无需了解子系统内部细节，大幅降低使用难度
- **降低耦合**：客户端代码与子系统解耦，子系统内部变化不影响客户端
- **简化使用**：将多个子系统调用的编排逻辑集中在一处，减少重复代码
- **分层架构支撑**：可为每一层子系统建立外观，形成清晰的层级边界

---

## 缺点

- 外观对象可能与所有子系统类耦合，演变成"神对象"（God Object）
- 可能掩盖子系统提供的高级能力，使高级用户无法充分利用底层功能
- 若外观承担过多职责，会成为系统的单点瓶颈
