# 享元模式 (Flyweight Pattern)

## 意图

运用共享技术有效支持大量细粒度对象，节省内存。

---

## 适用场景

- 程序需要生成数量巨大的相似对象，导致内存不足（如游戏中的粒子、子弹、树木）
- 对象中大部分状态可以提取为外在状态（由调用方传入），只有少量状态是内在的（可共享）
- 节省内存是首要目标，且愿意用少量 CPU 开销换取内存收益
- 移除外在状态后，相同对象可被多个场景复用

---

## 核心结构

```
Client
  │  传入 extrinsicState（外在状态）
  ▼
FlyweightFactory（工厂/缓存池）
  │  根据 intrinsicState 查找或创建
  ▼
Flyweight（享元对象）
  ├── intrinsicState（内在状态，共享，存于对象内部）
  └── operation(extrinsicState)（外在状态，每次由外部传入）
```

**角色说明：**

| 角色 | 职责 |
|------|------|
| `Flyweight` | 存储可在多个对象间共享的内在状态；通过方法参数接收外在状态 |
| `FlyweightFactory` | 管理享元对象缓存池，确保相同内在状态的对象只被创建一次 |
| `Client` | 维护或计算外在状态，通过工厂获取享元对象并传入外在状态调用 |

**内在状态 vs 外在状态：**

| 类型 | 定义 | 示例（子弹） |
|------|------|-------------|
| 内在状态 | 对象固有属性，独立于上下文，可共享 | 颜色、精灵图、伤害值 |
| 外在状态 | 依赖上下文，每个对象不同，不可共享 | 坐标、速度、方向 |

---

## 优点

- **显著节省内存**：大量相似对象共享内在状态，内存占用大幅降低
- **减少对象创建开销**：工厂缓存避免重复实例化相同对象

---

## 缺点

- 代码复杂度明显增加，需要区分并分离内在/外在状态
- 若外在状态计算或传递开销较大，可能反而增加 CPU 消耗
- 调试较困难，共享对象的状态变化会影响所有引用方
- 若对象种类繁多（内在状态各异），共享收益有限
